## Вступ

Сучасний ринок роздрібної торгівлі та швидкого харчування зазнає активної діджиталізації та автоматизації. Вулична торгівля кавою через мобільні кав'ярні (фургони кави, Coffee Vans) є яскравим прикладом динамічного мікробізнесу, який стрімко розвивається в умовах сучасних мегаполісів. Такі точки продажу мають власні унікальні логістичні та ресурсні обмеження: вони обмежені фізичною місткістю корисного простору фургону (в літрах) та бюджетом на закупівлю кавової продукції. Недотримання цих лімітів веде або до фізичної неможливості транспортування товару, або до заморожування обігових коштів підприємства та зниження загальної ефективності бізнесу.

**Актуальність розробки** спеціалізованої інформаційної системи для обліку товарів у фургоні кави зумовлена відсутністю на ринку простих, безкоштовних та автономних рішень складського обліку, які б враховували фізичні характеристики товарів (вагу, об'єм) та фінансові ліміти. Сучасні системи автоматизації ресторанного бізнесу (наприклад, Poster, iiko, r-keeper) є виключно хмарними та вимагають постійного Інтернет-з'єднання, що не завжди можливо на мобільних локаціях (парки, шосе, виїзні фестивалі), а також передбачають високу щомісячну абонентську плату, яка є нерентабельною для мікробізнесу на етапі старту. Розробка локального десктопного застосунку на базі мови Java та вбудованої СУБД SQLite повністю закриває цю нішу, надаючи надійне автономне рішення із нульовою вартістю володіння та високою надійністю.

**Об'єктом дослідження** у даній курсовій роботі є процеси автоматизації складського обліку, структурування, сортування за маржинальністю та оптимізації кавового асортименту в умовах обмеженого фізичного та фінансового ресурсу мобільного транспортного засобу.

**Предметом дослідження** є методи та архітектурні шаблони проектування програмного забезпечення, принципи об'єктно-орієнтованого програмування на мові Java, технології декларативного створення інтерфейсів користувача JavaFX та взаємодія з вбудованими реляційними базами даних за допомогою JDBC API.

**Метою роботи** є проектування та розробка повнофункціональної інформаційної системи «Coffee Van» для автоматизації процесів додавання, редагування, фільтрації, сортування за маржинальністю та аналізу асортименту кави, що завантажується у фургон кави із жорстким контролем фізичного об'єму та бюджету.

Для досягнення поставленої мети необхідно вирішити такі завдання:
1. Провести детальний аналіз предметної області та сформулювати функціональні й нефункціональні вимоги до системи.
2. Розробити об'єктно-орієнтовану модель предметної області, виділивши спільні характеристики кави та специфічні властивості її окремих підтипів.
3. Спроектувати реляційну схему бази даних SQLite для збереження інформації про товари та налаштування лімітів фургону.
4. Розробити графічний інтерфейс користувача на JavaFX із динамічною адаптацією форм під різні типи кави та реактивною фільтрацією в реальному часі.
5. Написати бізнес-логіку у вигляді сервісного шару та реалізувати збереження даних через JDBC DAO.
6. Забезпечити стійкість коду за допомогою модульного тестування JUnit 5 та автоматизованого UI-тестування TestFX.

## Розділ 1. Постановка задачі

### 1.1. Загальний опис предметної галузі та цілі системи
Логістика мобільної кав'ярні полягає в оптимізації завантаження транспортного засобу. Менеджер або водій-бариста перед виїздом на зміну має сформувати перелік товарів для продажу. Кава поставляється в різних упаковках і агрегатних станах: зернова, мелена, розчинна в банках та пакетиках. Кожен товар характеризується ціною за кілограм, вагою упаковки та її фізичним об'ємом у літрах.
Ціллю системи є надання зручного інструменту для ведення каталогу кави, що завантажена у фургон, з автоматичним відстеженням вільного об'єму та невитраченого бюджету кав'ярні.

### 1.2. Функціональні вимоги до програмної системи
На основі детального аналізу бізнес-процесів роздрібного продажу кави було висунуто такі функціональні вимоги до системи:
1. **Управління каталогом (CRUD):**
   - Додавання товарів з вибором типу кави та заповненням загальних та специфічних полів.
   - Редагування будь-яких полів доданих товарів.
   - Повне видалення окремого товару з каталогу фургону.
   - Швидке копіювання (дублювання) вибраного товару для прискорення заповнення схожими позиціями.
   - Скасування останньої операції видалення (Undo) за допомогою стеку операцій.
2. **Підтримка ієрархії кавової продукції:**
   - *Зернова кава:* зберігає комерційну назву, ціну, вагу, об'єм, тип упаковки, бал якості, а також унікальні атрибути: країну походження кавових зерен та ступінь їх обсмаження (light, medium, dark).
   - *Мелена кава:* додатково містить ступінь помолу (fine, medium, coarse).
   - *Розчинна у банках:* містить інформацію про наявність або відсутність додаткових гранул.
   - *Розчинна у пакетиках:* містить кількість окремих сашетів у коробці.
3. **Контроль та валідація обмежень фургону:**
   - Динамічний підрахунок поточного сумарного об'єму завантаженої кави та сумарної вартості.
   - Блокування додавання або оновлення товару, якщо це призведе до виходу за встановлені межі об'єму або бюджету фургону.
   - Зміна лімітів фургону через спеціальне діалогове вікно.
4. **Фільтрація та пошук в реальному часі:**
   - Пошук за назвою або підтипом кави.
   - Фільтр за типом кави (зернова, мелена тощо).
   - Фільтр за типом упаковки (пакет, мішок, коробка, банка, розсип).
   - Фільтрація за мінімальним балом якості (від 1 до 100).
   - Фільтрація за граничною ціною (пошук кави не дорожчої за вказане значення).
5. **Аналітичні функції:**
   - Сортування каталогу за відношенням ціни до ваги для визначення найбільш вигідних позицій.
   - Пошук товарів у заданому користувачем діапазоні якості.
   - Графічне представлення аналітики: кругова діаграма розподілу товарів за підтипами та стовпчикова діаграма вартості товарів.
6. **Імпорт/Експорт даних:**
   - Експорт поточного відфільтрованого списку товарів у формат CSV.
   - Імпорт товарів із зовнішнього файлу CSV з автоматичною перевіркою коректності записів та лімітів фургону.

### 1.3. Нефункціональні вимоги до системи
1. **Швидкодія та реактивність UI:**
   - Фільтрація та оновлення статусних панелей мають виконуватися миттєво при введенні символів користувачем (час відгуку до 100 мс).
2. **Надійність збереження даних:**
   - Реляційна цілісність бази даних має підтримуватися механізмом зовнішніх ключів (Foreign Keys).
   - Усі помилки доступу до даних мають детально логуватися. При виникненні критичних помилок (наприклад, пошкодження файлу БД) застосунок має надіслати email-сповіщення адміністратору та показати зрозумілу помилку користувачеві.
3. **Розширюваність архітектури:**
   - Програмна модель має дозволяти додавання нових типів кави (наприклад, кава в капсулах) із мінімальними змінами у коді.
4. **Середовище виконання:**
   - Робота під управлінням JDK 21 або вище, JavaFX 21+ та SQLite JDBC Driver.

## Розділ 2. Огляд літератури та обґрунтування технологій

### 2.1. Аналіз предметної галузі та схожих систем складського обліку
У сфері HoReCa (Hotel, Restaurant, Cafe) автоматизація є стандартом. Системи типу *Poster* або *iiko* пропонують хмарні рішення. Однак кавовий фургон часто працює в дорожніх умовах, на фестивалях або в парках, де мобільний інтернет може бути нестабільним або відсутнім. Хмарні додатки у разі втрати зв'язку блокують можливість проведення операцій. Окрім того, вартість підписки на такі сервіси (від $30 на місяць) є помітним фінансовим навантаженням для мікропідприємця. Це обґрунтовує доцільність створення локальної програми, що працює offline.

### 2.2. Порівняльний аналіз архітектурних підходів: десктоп проти веб
Перед початком проектування було детально проаналізовано два альтернативні підходи до розробки системи:

| Критерій порівняння | Веб-застосунок (SPA + REST API + PostgreSQL) | Десктоп-застосунок (Java + JavaFX + SQLite) |
| :--- | :--- | :--- |
| **Архітектура** | Трьохланкова (Клієнт - Сервер - База даних) | Одноланкова автономна (Вбудована БД) |
| **Залежність від мережі** | Стовідсоткова (не працює без Інтернету) | Відсутня (повністю автономна офлайн-робота) |
| **Складність розгортання** | Потребує налаштування серверного середовища | Запуск через один виконуваний файл JAR |
| **Інфраструктурні витрати** | Постійна оплата серверів, доменів, SSL | Нульова вартість використання локальних ресурсів |
| **Захист даних** | Дані передаються через мережу, є ризик перехоплення | Дані зберігаються локально на ПК власника |
| **Час відгуку UI** | Залежить від швидкості мережі та навантаження сервера | Мінімальний (миттєве локальне відображення) |

*Висновок:* для цільової аудиторії (поодинокі мобільні кав'ярні) десктопний варіант є найбільш раціональним, надійним та фінансово вигідним.

### 2.3. Мова програмування Java та її сучасні можливості
Мова **Java** є ідеальним вибором завдяки строгій типізації, кросплатформності та розвиненій екосистемі. Починаючи з версії Java 21, розробникам стали доступні сучасні мовні конструкції:
- **Switch Expressions:** дозволяють писати чистіший код без повторень ключового слова `break`.
- **Pattern Matching для instanceof та switch:** спрощує приведення типів при роботі з поліморфними списками об'єктів.
- **Records:** зручний спосіб оголошення незмінних класів-носіїв даних (DTO).
Ці технології підвищують безпеку коду на етапі компіляції та пришвидшують розробку.

### 2.4. Графічний стек JavaFX та реактивні колекції
**JavaFX** — це сучасний стандарт побудови GUI на Java, який замінив застарілий Swing. JavaFX має кілька ключових переваг:
1. **CSS стилізація:** дозволяє розділити логіку інтерфейсу та його зовнішній вигляд.
2. **Property Bindings (зв'язування властивостей):** дозволяє автоматично синхронізувати значення між компонентами UI та моделями.
3. **Реактивні обгортки ObservableList:** класи `FilteredList` та `SortedList` надають можливість накладати фільтри та правила сортування на списки даних декларативно. Коли базовий список змінюється (наприклад, додається товар у фоновому режимі), JavaFX автоматично перераховує фільтрацію та оновлює таблицю `TableView` без примусового перезавантаження.

### 2.5. СУБД SQLite: архітектура та порівняння з альтернативами
**SQLite** — це вбудована реляційна СУБД. На відміну від серверних баз даних (PostgreSQL, MySQL), SQLite не потребує встановлення сервера, налаштування портів та користувачів. База даних зберігається у звичайному локальному файлі `coffee_van.db`. При цьому SQLite підтримує стандарти SQL-92, транзакційність (ACID) та каскадне видалення через зовнішні ключі, що робить її найкращим вибором для локальних десктопних програм.

### 2.6. Архітектурний шаблон MVC та принципи проектування SOLID
У проекті використано архітектурний шаблон **Model-View-Controller (MVC)**:
- **Model:** пакети `model` та `dao` (класи `Coffee` та підкласи, `Van`, `DatabaseManager`, класи DAO).
- **View:** пакет `ui` (класи вікон та діалогів).
- **Controller:** сервісний шар `VanService` та логіка подій в `MainWindow`.

При проектуванні дотримано принципів **SOLID**:
- *Single Responsibility Principle (Принцип єдиної відповідальності):* `DatabaseManager` займається виключно базою даних, `CoffeeDaoImpl` — збереженням кави, `SmtpEmailNotifier` — відправкою листів.
- *Open/Closed Principle (Принцип відкритості/закритості):* ієрархія класів кави відкрита для створення нових підкласів, при цьому базовий клас `Coffee` не потребує модифікацій.
- *Liskov Substitution Principle (Принцип підстановки Ліскова):* будь-яка бізнес-логіка в `VanService` працює з типом `Coffee` поліморфно, не ламаючи поведінку програми при заміні на `BeanCoffee` чи `GroundCoffee`.

### 2.7. Обґрунтування системи тестування: JUnit 5 та TestFX
Якість програмного продукту забезпечується автоматизованими тестами. **JUnit 5** надає сучасний інструментарій для юніт-тестів. Для ізоляції тестів бази даних від локального файлу використовується in-memory підключення (`jdbc:sqlite::memory:`), завдяки чому тестова БД створюється в оперативній пам'яті перед кожним тестом і знищується після нього.
**TestFX** дозволяє симулювати реальні дії користувача в інтерфейсі (кліки, введення тексту), що дає змогу перевірити коректність роботи форм та валідаторів JavaFX в автоматичному режимі.

### 3.1. Об'єктно-орієнтована декомпозиція предметної галузі
Предметна область мобільної кав'ярні була декомпонована на набір класів:
1. **Абстрактний клас `Coffee`:** базовий об'єкт, який описує загальні властивості будь-якого кавового продукту:
   - `id` (int): унікальний ідентифікатор в БД.
   - `name` (String): найменування товару.
   - `pricePerKg` (double): ціна за 1 кг кави (грн).
   - `weightKg` (double): вага нетто однієї упаковки (кг).
   - `volumeLiters` (double): об'єм, який займає упаковка у фургоні (л).
   - `packagingType` (PackagingType): тип упаковки (enum).
   - `qualityScore` (int): оцінка якості кави від 1 до 100.
   - Метод `getTotalPrice()`: повертає ціну за упаковку (`pricePerKg * weightKg`).
   - Метод `getPriceToWeightRatio()`: відношення ціни за кг до ваги для оцінки маржинальності.
2. **Клас `BeanCoffee`:** наслідує `Coffee`. Додає атрибути:
   - `origin` (String): країна вирощування зерен.
   - `roastLevel` (String): ступінь обсмаження.
3. **Клас `GroundCoffee`:** наслідує `Coffee`. Додає атрибут:
   - `grindSize` (String): розмір помолу.
4. **Клас `InstantCoffeeJar`:** наслідує `Coffee`. Додає атрибут:
   - `granules` (int): прапорець гранульованості (1 — так, 0 — ні).
5. **Клас `InstantCoffeeSachet`:** наслідує `Coffee`. Додає атрибут:
   - `sachetsCount` (int): кількість пакетиків-сашетів.
6. **Enum `PackagingType`:** містить значення `PACKAGE`, `BAG`, `BOX`, `JAR`, `BULK`.
7. **Клас `Van`:** модель фургону, що містить поточні значення завантаження та обмеження.
8. **Клас `VanService`:** сервісний шар бізнес-логіки.

### 3.2. Архітектура системи за шаблоном MVC
Потік даних у системі організовано наступним чином:
`MainWindow` (View) відображає дані та зчитує команди користувача. При натисканні кнопок View викликає відповідні методи `VanService` (Controller). `VanService` звертається до `CoffeeDao` для виконання CRUD операцій. Отримані результати автоматично оновлюють реактивний список `masterList` (ObservableList), що миттєво відображається в `TableView`.

### 3.3. Проектування схеми реляційної бази даних
Для реалізації об'єктного спадкування в реляційній базі даних було обрано паттерн **Single Table Inheritance** (одна таблиця на всю ієрархію). Це спрощує структуру бази даних та підвищує швидкість вибірки всього каталогу.

#### Таблиця `coffee`
```sql
CREATE TABLE IF NOT EXISTS coffee (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    coffee_type   TEXT    NOT NULL,
    name          TEXT    NOT NULL,
    price_per_kg  REAL    NOT NULL,
    weight_kg     REAL    NOT NULL,
    volume_liters REAL    NOT NULL,
    packaging     TEXT    NOT NULL,
    quality_score INTEGER NOT NULL,
    extra1        TEXT,
    extra2        TEXT,
    extra3        TEXT
);
```
*Правила використання динамічних полів `extra`:*
* Для `Зернова`: `extra1` = origin, `extra2` = roastLevel, `extra3` = NULL
* Для `Мелена`: `extra1` = grindSize, `extra2` = NULL, `extra3` = NULL
* Для `Розчинна (банка)`: `extra1` = granules (як рядок), `extra2` = NULL, `extra3` = NULL
* Для `Розчинна (пакетик)`: `extra1` = sachetsCount (як рядок), `extra2` = NULL, `extra3` = NULL

#### Таблиця `van_settings`
Таблиця містить налаштування лімітів фургону. Вона завжди містить лише один рядок з `id = 1`.
```sql
CREATE TABLE IF NOT EXISTS van_settings (
    id            INTEGER PRIMARY KEY,
    max_volume    REAL    NOT NULL DEFAULT 1000.0,
    max_budget    REAL    NOT NULL DEFAULT 50000.0
);
```

## Розділ 4. Програмне рішення

### 4.1. Взаємодія з базою даних через JDBC

#### 4.1.1. Ініціалізація з'єднання та схеми БД
Клас `DatabaseManager` реалізований за шаблоном **Singleton** для запобігання паралельним блокуванням файлу SQLite. Підключення ініціалізується один раз:
```java
connection = DriverManager.getConnection("jdbc:sqlite:coffee_van.db");
```
Для забезпечення реляційної цілісності даних при кожному підключенні обов'язково виконується PRAGMA-налаштування:
```java
try (Statement stmt = connection.createStatement()) {
    stmt.execute("PRAGMA foreign_keys = ON;");
}
```
Метод `initSchema()` перевіряє наявність таблиць `coffee` та `van_settings`, автоматично створюючи їх при першому запуску програми.

#### 4.1.2. Параметризовані запити та управління транзакціями
Всі SQL-запити до бази даних виконуються через `PreparedStatement` для запобігання атакам типу SQL-ін'єкцій та оптимізації швидкодії.
При додаванні товару `CoffeeDaoImpl` зчитує згенерований SQLite автоінкрементний ключ та записує його в об'єкт:
```java
ResultSet keys = ps.getGeneratedKeys();
if (keys.next()) {
    coffee.setId(keys.getInt(1));
}
```
У разі масового імпорту з CSV або повного очищення фургону операції обгортаються у транзакції (`connection.setAutoCommit(false)`), що забезпечує атомарність та прискорює виконання записів у десятки разів.

### 4.2. Реалізація графічного інтерфейсу на JavaFX

#### 4.2.1. Структура головного вікна застосунку
Графічний інтерфейс побудований повністю програмним шляхом без використання FXML, що дозволило досягти максимального контролю над створенням компонентів.
Головне вікно (`MainWindow`) спроектоване на основі `BorderPane`. Верхня частина містить брендовану панель заголовка з кнопкою швидких налаштувань. Центральна частина містить інформаційні картки (статистичні картки), панель інструментів швидкого доступу до CRUD операцій та таблицю `TableView` для списку товарів. Справа динамічно відображається панель детального перегляду вибраного товару. Нижній слот відведений під статус-бар.

#### 4.2.2. Панель каталогу та система фільтрації в реальному часі
Реактивна фільтрація реалізована за допомогою обгортки над списком товарів:
```java
masterList = FXCollections.observableArrayList();
filteredList = new FilteredList<>(masterList, p -> true);
SortedList<Coffee> sortedList = new SortedList<>(filteredList);
sortedList.comparatorProperty().bind(tableView.comparatorProperty());
tableView.setItems(sortedList);
```
Кожен елемент фільтрації (пошуковий рядок, ComboBox типу, фільтр максимальної ціни, повзунок якості) підписаний на подію зміни значення (`addListener`), яка миттєво викликає метод `applyActiveFilters()`. Метод будує спільний `Predicate`, що комбінує всі активні правила за логікою AND.

#### 4.2.3. Динамічний діалог додавання кави та валідація
Діалогове вікно `CoffeeDialog` динамічно перебудовує форму вводу залежно від обраного у випадаючому списку типу кави. При виборі «Зернова» з'являються текстові поля для країни походження та рівня обсмаження, а при виборі «Мелена» — поле для ступеня помолу.
Кнопка збереження ("OK") підв'язана до валідатора за допомогою реактивного зв'язування (`Bindings`): вона залишається неактивною, доки користувач не введе коректні числові дані (ціна, вага, об'єм мають бути більшими за нуль) та не заповнить назву товару.

```java
package ua.lpnu.coffevan.model;

import java.util.Objects;

public abstract class Coffee {
    private int id;
    private String name;
    private double pricePerKg;
    private double weightKg;
    private double volumeLiters;
    private PackagingType packagingType;
    private int qualityScore;

    protected Coffee() {}

    protected Coffee(String name, double pricePerKg, double weightKg,
                     double volumeLiters, PackagingType packagingType, int qualityScore) {
        this.name = name;
        this.pricePerKg = pricePerKg;
        this.weightKg = weightKg;
        this.volumeLiters = volumeLiters;
        this.packagingType = packagingType;
        this.qualityScore = qualityScore;
    }

    public abstract String getCoffeeType();

    public double getTotalPrice() {
        return pricePerKg * weightKg;
    }

    public double getPriceToWeightRatio() {
        if (weightKg == 0) return 0;
        return pricePerKg / weightKg;
    }

    // Getters and Setters...
    public int getId() { return id; }
    public void setId(int id) { this.id = id; }
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public double getPricePerKg() { return pricePerKg; }
    public void setPricePerKg(double pricePerKg) { this.pricePerKg = pricePerKg; }
    public double getWeightKg() { return weightKg; }
    public void setWeightKg(double weightKg) { this.weightKg = weightKg; }
    public double getVolumeLiters() { return volumeLiters; }
    public void setVolumeLiters(double volumeLiters) { this.volumeLiters = volumeLiters; }
    public PackagingType getPackagingType() { return packagingType; }
    public void setPackagingType(PackagingType packagingType) { this.packagingType = packagingType; }
    public int getQualityScore() { return qualityScore; }
    public void setQualityScore(int qualityScore) { this.qualityScore = qualityScore; }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Coffee coffee)) return false;
        return id == coffee.id && Double.compare(coffee.pricePerKg, pricePerKg) == 0 &&
               Double.compare(coffee.weightKg, weightKg) == 0 && qualityScore == coffee.qualityScore &&
               Objects.equals(name, coffee.name);
    }

    @Override
    public int hashCode() {
        return Objects.hash(id, name, pricePerKg, weightKg, qualityScore);
    }
}
```

### Додаток Б. Реалізація DAO для SQLite (`CoffeeDaoImpl.java`)
```java
package ua.lpnu.coffevan.dao;

import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;
import ua.lpnu.coffevan.model.*;
import java.sql.*;
import java.util.ArrayList;
import java.util.List;
import java.util.Optional;

public class CoffeeDaoImpl implements CoffeeDao {
    private static final Logger logger = LogManager.getLogger(CoffeeDaoImpl.class);
    private final Connection connection;

    public CoffeeDaoImpl(Connection connection) {
        this.connection = connection;
    }

    @Override
    public int save(Coffee coffee) {
        String sql = "INSERT INTO coffee (coffee_type, name, price_per_kg, weight_kg, volume_liters, packaging, quality_score, extra1, extra2, extra3) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?)";
        try (PreparedStatement ps = connection.prepareStatement(sql, Statement.RETURN_GENERATED_KEYS)) {
            fillCommonFields(ps, coffee);
            ps.executeUpdate();
            ResultSet keys = ps.getGeneratedKeys();
            if (keys.next()) {
                int id = keys.getInt(1);
                coffee.setId(id);
                logger.info("Saved coffee id={}", id);
                return id;
            }
        } catch (SQLException e) {
            logger.error("Error saving coffee", e);
        }
        return -1;
    }

    @Override
    public List<Coffee> findAll() {
        List<Coffee> list = new ArrayList<>();
        String sql = "SELECT * FROM coffee";
        try (Statement stmt = connection.createStatement();
             ResultSet rs = stmt.executeQuery(sql)) {
            while (rs.next()) {
                list.add(mapRow(rs));
            }
        } catch (SQLException e) {
            logger.error("Error loading all items", e);
        }
        return list;
    }

    private void fillCommonFields(PreparedStatement ps, Coffee coffee) throws SQLException {
        ps.setString(1, coffee.getCoffeeType());
        ps.setString(2, coffee.getName());
        ps.setDouble(3, coffee.getPricePerKg());
        ps.setDouble(4, coffee.getWeightKg());
        ps.setDouble(5, coffee.getVolumeLiters());
        ps.setString(6, coffee.getPackagingType().name());
        ps.setInt(7, coffee.getQualityScore());

        if (coffee instanceof BeanCoffee bc) {
            ps.setString(8, bc.getOrigin());
            ps.setString(9, bc.getRoastLevel());
            ps.setNull(10, Types.VARCHAR);
        } else if (coffee instanceof GroundCoffee gc) {
            ps.setString(8, gc.getGrindSize());
            ps.setNull(9, Types.VARCHAR);
            ps.setNull(10, Types.VARCHAR);
        } else if (coffee instanceof InstantCoffeeJar ij) {
            ps.setString(8, String.valueOf(ij.getGranules()));
            ps.setNull(9, Types.VARCHAR);
            ps.setNull(10, Types.VARCHAR);
        } else if (coffee instanceof InstantCoffeeSachet is) {
            ps.setString(8, String.valueOf(is.getSachetsCount()));
            ps.setNull(9, Types.VARCHAR);
            ps.setNull(10, Types.VARCHAR);
        }
    }

    private Coffee mapRow(ResultSet rs) throws SQLException {
        String type = rs.getString("coffee_type");
        String name = rs.getString("name");
        double price = rs.getDouble("price_per_kg");
        double weight = rs.getDouble("weight_kg");
        double volume = rs.getDouble("volume_liters");
        PackagingType packaging = PackagingType.valueOf(rs.getString("packaging"));
        int quality = rs.getInt("quality_score");
        String extra1 = rs.getString("extra1");
        String extra2 = rs.getString("extra2");

        Coffee c = switch (type) {
            case "Зернова" -> new BeanCoffee(name, price, weight, volume, packaging, quality, extra1, extra2);
            case "Мелена" -> new GroundCoffee(name, price, weight, volume, packaging, quality, extra1);
            case "Розчинна (банка)" -> new InstantCoffeeJar(name, price, weight, volume, quality, Integer.parseInt(extra1));
            case "Розчинна (пакетик)" -> new InstantCoffeeSachet(name, price, weight, volume, quality, Integer.parseInt(extra1));
            default -> throw new IllegalStateException("Unknown coffee type: " + type);
        };
        c.setId(rs.getInt("id"));
        return c;
    }
}
```

### Додаток В. Реалізація бізнес-сервісу (`VanService.java`)
```java
package ua.lpnu.coffevan.service;

import org.apache.logging.log4j.LogManager;
import org.apache.logging.log4j.Logger;
import ua.lpnu.coffevan.dao.CoffeeDao;
import ua.lpnu.coffevan.dao.VanSettingsDaoInterface;
import ua.lpnu.coffevan.model.Coffee;
import ua.lpnu.coffevan.model.Van;

import java.util.Comparator;
import java.util.List;
import java.util.stream.Collectors;

public class VanService {
    private static final Logger logger = LogManager.getLogger(VanService.class);
    private final CoffeeDao coffeeDao;
    private final VanSettingsDaoInterface vanSettingsDao;
    private Van van;

    public VanService(CoffeeDao coffeeDao, VanSettingsDaoInterface vanSettingsDao) {
        this.coffeeDao = coffeeDao;
        this.vanSettingsDao = vanSettingsDao;
        this.van = vanSettingsDao.load();
        recalculateVanUsage();
    }

    private void recalculateVanUsage() {
        double totalVolume = 0;
        double totalBudget = 0;
        for (Coffee c : coffeeDao.findAll()) {
            totalVolume += c.getVolumeLiters();
            totalBudget += c.getTotalPrice();
        }
        van.setUsedVolumeLiters(totalVolume);
        van.setSpentBudget(totalBudget);
    }

    public boolean addCoffee(Coffee coffee) {
        if (!van.canLoad(coffee)) {
            logger.warn("Cannot load coffee '{}': van capacity or budget exceeded", coffee.getName());
            return false;
        }
        coffeeDao.save(coffee);
        van.load(coffee);
        logger.info("Added coffee '{}' to van", coffee.getName());
        return true;
    }

    public void removeCoffee(Coffee coffee) {
        coffeeDao.delete(coffee.getId());
        van.unload(coffee);
        logger.info("Removed coffee id={} from van", coffee.getId());
    }

    public void updateCoffee(Coffee original, Coffee updated) {
        van.unload(original);
        coffeeDao.update(updated);
        van.load(updated);
        logger.info("Updated coffee id={}", updated.getId());
    }

    public List<Coffee> getAllCoffee() {
        return coffeeDao.findAll();
    }

    public List<Coffee> sortByPriceToWeightRatio() {
        return coffeeDao.findAll().stream()
                .sorted(Comparator.comparingDouble(Coffee::getPriceToWeightRatio))
                .collect(Collectors.toList());
    }

    public List<Coffee> findByQualityRange(int minQuality, int maxQuality) {
        return coffeeDao.findAll().stream()
                .filter(c -> c.getQualityScore() >= minQuality && c.getQualityScore() <= maxQuality)
                .collect(Collectors.toList());
    }

    public void clearVan() {
        coffeeDao.deleteAll();
        van.setUsedVolumeLiters(0);
        van.setSpentBudget(0);
        logger.info("Van cleared");
    }

    public Van getVan() { return van; }

    public void updateVanSettings(double maxVolume, double maxBudget) {
        van.setMaxVolumeLiters(maxVolume);
        van.setMaxBudget(maxBudget);
        vanSettingsDao.save(van);
        logger.info("Van settings updated");
    }

    public double getTotalVolume() { return van.getUsedVolumeLiters(); }
    public double getTotalBudget() { return van.getSpentBudget(); }
}
```
